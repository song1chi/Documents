## 2. 인증·권한

DeepCube.AI는 엔터프라이즈급 보안을 위한 포괄적인 인증 및 권한 관리 체계를 제공합니다. JWT 기반의 토큰 인증, 세분화된 RBAC(Role-Based Access Control), API Key 관리 등을 통해 안전하고 유연한 접근 제어가 가능합니다.

### 주요 기능

**A-01. 사용자 로그인**

사용자 인증을 위한 기본 로그인 기능을 제공합니다. ID와 비밀번호를 입력받아 세션을 생성하고 JWT 토큰을 발급합니다. REST API를 통해 액세스 토큰과 리프레시 토큰을 함께 제공하여 안전한 세션 관리를 지원합니다. 기업 환경에서는 SSO(Single Sign-On)나 LDAP와의 연동 필요 여부를 사전에 검토해야 합니다. 다단계 인증(MFA)이나 소셜 로그인과의 통합도 협의를 통해 확장할 수 있습니다.

**A-02. 토큰 갱신**

액세스 토큰 만료 시 리프레시 토큰으로 새 액세스 토큰을 재발급합니다. REST API를 통해 사용자가 재로그인 없이 작업을 이어갈 수 있도록 합니다. 세션 유지 정책(유휴 시간, 최대 세션 수 등)을 조직 보안 정책에 맞게 설정하는 것이 중요합니다. 리프레시 토큰 회전(rotation) 정책을 적용하면 토큰 탈취 위험을 줄일 수 있습니다.

**A-03. 로그아웃**

리프레시 토큰을 폐기하고 세션을 종료합니다. REST API를 통해 클라이언트와 서버 양쪽에서 세션 상태를 정리합니다. 동시 세션 관리나 관리자에 의한 강제 로그아웃 요구가 있는 환경에서는 정책을 사전에 정의해야 합니다. 공용 단말 사용 환경에서는 로그아웃 시 로컬 캐시 정리도 함께 고려합니다.

**A-04. 연동 토큰 발급**

외부 시스템 연동을 위해 API Key를 JWT 토큰으로 교환합니다. REST API를 통해 서버 간(B2B) 인증을 수행하며, 사용자 세션과 분리된 연동 전용 토큰을 발급합니다. API Key의 발급·보관·회전·폐기 정책을 운영 절차에 포함해야 합니다. 키 유출 시 즉시 폐기하고 재발급할 수 있는 체계를 마련하는 것이 좋습니다.

**A-05. 토큰 검증**

발급된 토큰의 유효성을 검증합니다. REST API를 통해 토큰 서명, 만료 시간, 권한 범위를 확인합니다. API 게이트웨이와 연계하여 보호된 리소스에 대한 1차 인증 차단을 구현할 수 있습니다. 엣지 인증과 애플리케이션 인증의 역할 분담을 명확히 하는 것이 운영에 유리합니다.

**A-06. 사용자 정보·권한 조회**

로그인한 사용자의 프로필과 부여된 권한(grants)을 조회합니다. REST API를 통해 메뉴 노출, 기능 접근, 시나리오 실행 범위 등을 클라이언트에 전달합니다. 조직별 메뉴·기능 노출 정책을 RBAC 설정과 일치시켜야 합니다. 권한 변경 후 UI 반영 시점(즉시 vs 갱신 시)도 함께 정의합니다.

**A-07. RBAC — 사용자·그룹**

사용자와 그룹을 생성·조회·수정·삭제하고, 그룹 멤버십을 관리합니다. 관리 UI와 API를 통해 조직 구조를 반영한 권한 체계를 구축할 수 있습니다. 부서·팀·프로젝트 단위 그룹 설계가 도입 초기에 중요합니다. 외부 인사 시스템과의 동기화 필요 여부도 검토합니다.

**A-08. RBAC — 메뉴·기능 Grant**

메뉴와 기능 단위로 접근 권한을 부여합니다. 관리 UI와 API를 통해 최소 권한 원칙에 맞게 세분화된 제어가 가능합니다. 관리자·설계자·운영자·일반 사용자 등 역할별 메뉴 구성을 사전에 정의해야 합니다. 기능 단위 권한은 감사 요구와 직결되므로 변경 이력 관리를 권장합니다.

**A-09. RBAC — 시나리오·카테고리 Grant**

시나리오와 카테고리 단위로 접근을 제어합니다. 관리 UI와 API를 통해 부서별·업무별 시나리오 분리가 가능합니다. 민감 업무 시나리오는 별도 카테고리로 격리하고 접근 그룹을 제한하는 구성이 일반적입니다. 공유 시나리오와 개인 시나리오의 권한 모델을 구분하여 운영합니다.

**A-10. 권한 변경 동기화**

권한이 변경되면 클라이언트에 변경 사항을 반영합니다. 토큰 버전 관리를 통해 권한 회수·변경을 추적합니다. 즉시 무효화와 토큰 갱신 시 반영 중 어떤 방식을 채택할지 보안 정책에 맞게 선택해야 합니다. 퇴사·부서 이동 등 인사 변경 시 권한 반영 SLA를 정의하는 것이 좋습니다.

**A-11. 라이선스 관리**

제품 라이선스를 검증하고 설정합니다. 관리 UI와 API를 통해 상용 배포 시 라이선스 정책을 적용할 수 있습니다. 사용자 수, 기능 범위, 만료일 등 라이선스 조건을 계약과 일치시켜야 합니다. 라이선스 위반 시 동작(경고 vs 차단)도 사전에 합의합니다.

**A-12. 게이트웨이 인증 연동**

API 게이트웨이와 인증 체계를 통합합니다. 게이트웨이 플러그인을 통해 보호 경로에 대한 1차 인증을 수행할 수 있습니다. 고객사의 기존 게이트웨이·WAF·SSO 인프라와의 연계 방식을 도입 전에 확인해야 합니다. 내부 서비스 간 통신과 외부 연동 통신의 인증 정책을 분리하는 것이 안전합니다.

## 3. 시나리오 관리

시나리오 관리 기능은 AI 워크플로를 조직·업무 단위로 분류하고, 사용자별로 접근·실행·공유할 수 있게 합니다. 테스트(beta)와 운영(prod) 환경을 분리하여 안전하게 배포·운영할 수 있습니다.

### 주요 기능

**B-01. 시나리오 카테고리**

폴더형 카테고리로 시나리오를 분류합니다. 관리 UI와 API를 통해 조직·업무별 분류 체계를 구성할 수 있습니다. 카테고리 구조는 초기에 단순하게 시작하고, 시나리오 수가 늘어나면 계층을 확장하는 방식이 효과적입니다. 카테고리별 권한 부여와 연계하여 부서별 격리도 가능합니다.

**B-02. 시나리오 목록(카테고리)**

카테고리를 선택하면 해당 범주의 시나리오 목록을 조회합니다. UI와 API를 통해 사용자별 노출 범위가 RBAC에 따라 제한됩니다. 목록 화면에서 검색·정렬·필터 기능이 UX에 큰 영향을 줍니다. 대량 시나리오 환경에서는 카테고리 설계가 탐색 효율을 좌우합니다.

**B-03. 내 시나리오**

현재 사용자에게 권한이 부여된 전체 시나리오를 한눈에 조회합니다. UI와 API를 통해 연동·운영 대상 시나리오를 빠르게 파악할 수 있습니다. 프로젝트 초기에는 검증 대상 시나리오 1~3건을 명확히 지정하는 것이 좋습니다. 권한 없는 시나리오는 목록에 노출되지 않습니다.

**B-04. 즐겨찾기**

자주 사용하는 시나리오를 북마크하여 빠르게 접근합니다. UI를 통해 개인별 단축 목록을 관리합니다. 반복 업무가 많은 사용자에게 생산성 향상 효과가 큽니다. 조직 공통 즐겨찾기와 개인 즐겨찾기의 역할을 구분할지 검토합니다.

**B-05. 공유 시나리오**

다른 사용자가 공유한 시나리오를 조회합니다. UI를 통해 협업·템플릿 공유가 가능합니다. 공유 범위(개인·그룹·전체)와 편집 권한을 정책으로 정의해야 합니다. 공유 시나리오의 버전·출처 추적이 필요한 환경에서는 감사 로그와 연계합니다.

**B-06. 시나리오 실행(UI)**

관리 화면에서 프롬프트를 입력하고 시나리오를 실행한 뒤 결과를 표시합니다. UI를 통해 비기술 사용자도 AI 워크플로를 직접 사용할 수 있습니다. 응답 시간 SLA와 결과 표시 형식(텍스트·표·파일)을 업무 요구에 맞게 설계합니다. 긴 실행 시간이 예상되면 SSE 스트리밍과 연계합니다.

**B-07. beta/prod 배포**

시나리오를 테스트(beta)와 운영(prod) 환경에 분리 배포합니다. 관리 UI와 API를 통해 환경별 배포 상태를 관리합니다. 테스트 환경에서 검증 완료 후 운영 환경으로 승격하는 절차를 정의하는 것이 안전합니다. 운영 배포 시 롤백 절차도 함께 마련합니다.

**B-08. 시나리오 편집 진입**

관리 화면에서 비주얼 에디터로 이동하여 시나리오를 편집합니다. UI를 통해 편집 권한이 있는 사용자만 설계 화면에 접근합니다. 편집·저장·배포 이벤트는 감사 로그에 기록하는 것을 권장합니다. 편집 중인 시나리오와 운영 중인 시나리오의 충돌을 방지하는 정책이 필요합니다.

**B-09. 연동 시나리오 목록 API**

B2B 클라이언트가 호출 가능한 시나리오 목록을 REST API로 제공합니다. 외부 시스템 연동 시 대상 시나리오 ID를 사전에 확정해야 합니다. 사용 중(inUse) 상태인 시나리오만 노출하는 등 필터 조건을 연동 규격에 맞게 정의합니다. 목록 변경 시 외부 시스템에 영향이 없도록 버전·호환성을 관리합니다.

## 4. 시나리오 설계

시나리오 설계 기능은 노드·엣지 기반 비주얼 에디터로 AI 워크플로를 구성합니다. 설계자는 노드를 배치하고 연결하여 LLM, RAG, MCP, 서브시나리오 등을 조합한 복합 업무 흐름을 만들 수 있습니다.

### 주요 기능

**C-01. 시각적 캔버스**

노드와 엣지를 드래그·연결하는 비주얼 편집 환경을 제공합니다. 비주얼 에디터를 통해 코드 없이 AI 프로세스를 설계할 수 있습니다. 설계자 교육·온보딩 계획이 도입 성공에 중요합니다. 복잡한 시나리오는 서브시나리오로 모듈화하여 가독성을 유지합니다.

**C-02. 노드 풀**

사용 가능한 노드 타입과 설정 메타데이터를 에디터에 제공합니다. 에디터 UI를 통해 프로젝트에 필요한 노드(LLM, RAG, MCP 등)가 포함되어 있는지 확인합니다. 커스텀 노드 확장이 필요한 경우 노드 메타 동기화와 함께 검토합니다. 노드별 입력·출력 스키마를 설계 단계에서 문서화하면 운영이 수월합니다.

**C-03. 시나리오 CRUD**

시나리오를 생성·조회·수정·삭제합니다. REST API를 통해 프로그래밍 방식의 관리도 가능합니다. 버전 관리·백업 정책을 정의하여 실수로 인한 손실을 방지합니다. 삭제 시 운영 배포 여부를 확인하는 안전 장치를 권장합니다.

**C-04. 시나리오 목록(에디터)**

설계자용 전체 시나리오 목록을 에디터에서 조회합니다. 에디터 UI와 API를 통해 관리 화면과 다른 관점의 목록을 제공할 수 있습니다. 대량 시나리오 환경에서는 검색·태그·카테고리 연계가 필요합니다.

**C-05. 노드 메타 동기화**

노드 정의를 데이터베이스와 동기화합니다. REST API를 통해 노드 카탈로그를 최신 상태로 유지합니다. 커스텀 노드 확장 시 이 동기화 절차가 필수입니다. 동기화 실패 시 에디터에 노드가 누락될 수 있으므로 운영 모니터링을 권장합니다.

**C-06. MCP 에이전트 목록**

등록된 MCP 에이전트와 사용 가능한 도구 목록을 조회합니다. REST API를 통해 에디터에서 MCP 노드 설정 시 참조합니다. MCP 서버 구축·등록이 선행되어야 합니다. 도구 목록 변경 시 캐시 갱신 정책을 확인합니다.

**C-07. 단일 노드 테스트 실행**

에디터에서 개별 노드를 실행하여 동작을 검증합니다. REST API와 SSE를 통해 결과를 실시간으로 확인할 수 있습니다. 전체 시나리오 실행 전 디버깅·검증에 활용합니다. LLM·MCP 노드는 외부 의존성이 있으므로 테스트 환경 API 키·연결 정보를 별도 관리합니다.

**C-08. 관리 UI 연동**

관리 화면에서 에디터로 시나리오 메타(식별자, 사용자, 편집 가능 여부 등)를 전달합니다. 내부 연동을 통해 세션·권한 정보가 에디터에 일관되게 전달됩니다. SSO·세션 전달 방식을 고객 인증 체계에 맞게 구성합니다. iframe 또는 리다이렉트 방식의 보안 정책도 함께 검토합니다.

## 5. 워크플로 실행

워크플로 실행 엔진은 설계된 시나리오를 실제로 수행합니다. 다양한 노드 타입을 순차·병렬로 실행하고, 비동기 작업 큐와 SSE 스트리밍으로 긴 작업의 진행 상황을 전달합니다.

### 5.1 노드 타입

워크플로는 여러 종류의 노드를 조합하여 구성합니다. `start` 노드에서 입력을 받고, 중간 노드에서 LLM·검색·연동을 수행한 뒤 `end` 노드에서 결과를 반환하는 흐름이 일반적입니다.

**start**

시나리오의 시작점으로, 외부에서 전달된 입력 파라미터를 수신합니다. 입력 데이터 구조(질문, 컨텍스트, 파일 참조 등)를 설계 단계에서 명확히 정의해야 합니다. B2B 연동 시 API 요청 본문과 start 노드 입력 매핑을 사전에 합의합니다.

**end**

시나리오의 종료점으로, 최종 결과를 호출자에게 반환합니다. 출력 형식(텍스트, JSON, 구조화 객체)을 업무 요구에 맞게 정의합니다. 다중 end 노드나 조건부 종료 패턴도 구성할 수 있습니다.

**llm**

대규모 언어 모델을 호출하여 프롬프트를 처리합니다. 모델 선택, API 키, 토큰 비용, 응답 지연이 주요 고려사항입니다. 여러 LLM 프로바이더를 노드별로 지정할 수 있으며, 폐쇄망에서는 자체 호스팅 모델을 사용합니다.

**rag**

벡터 데이터베이스에서 관련 문서를 검색하고 컨텍스트로 제공합니다. 인덱스·컬렉션 사전 준비가 필요합니다. 검색 품질은 임베딩 모델, 청크 크기, 문서 품질에 크게 좌우됩니다. MCP와 병행하여 문서와 실시간 데이터를 함께 활용할 수 있습니다.

**websearch**

외부 웹 검색 API를 호출하여 최신 정보를 통합합니다. 외부망 연결과 검색 API 키가 필요합니다. 검색 결과의 신뢰성·출처 표시 정책을 업무 규정에 맞게 정의합니다.

**textinput**

텍스트 템플릿과 변수 치환을 수행합니다. 이전 노드 출력이나 입력 파라미터를 문자열에 삽입하는 데 사용합니다. 프롬프트 조립·포맷팅 단계로 자주 활용됩니다.

**scenario**

다른 시나리오를 서브시나리오로 호출합니다. 재사용 가능한 모듈화된 워크플로 설계가 가능합니다. 공통 전처리·후처리 로직을 서브시나리오로 분리하면 유지보수가 용이합니다.

**runtime_scenario**

런타임에 동적으로 시나리오를 생성·실행하는 고급 패턴을 지원합니다. 가변 업무 흐름이나 에이전트형 자율 실행에 활용됩니다. 설계·테스트 복잡도가 높으므로 필요 시에만 도입을 권장합니다.

**mcp_agent**

MCP(Model Context Protocol)를 통해 데이터베이스·업무 도구에 직접 연동합니다. 대상 시스템 MCP 서버 구성이 핵심 선행 조건입니다. 문서 RAG 없이도 실시간·구조화 데이터 기반 Q&A와 업무 처리가 가능한 핵심 기능입니다.

**mcp**

MCP를 동기 방식으로 직접 호출합니다. 단순·짧은 도구 호출에 적합하나, 처리 시간과 동시성 제약을 고려해야 합니다. 장시간 작업은 mcp_agent 노드 사용을 권장합니다.

### 5.2 실행 기능

**D-11. 시나리오 비동기 실행**

전체 시나리오를 비동기로 실행하고 작업 ID를 반환합니다. REST API(비동기)를 통해 호출자는 즉시 응답을 받고, SSE로 결과를 수신합니다. 타임아웃·재시도 정책을 외부 연동 규격에 맞게 정의합니다. 동시 실행 수와 리소스 한도를 모니터링합니다.

**D-12. 단일 노드 실행**

개별 노드만 실행하여 테스트·디버깅합니다. REST API를 통해 에디터와 연동됩니다. 전체 시나리오 실행 전 노드 단위 검증에 활용합니다.

**D-13. 동기 단순 쿼리**

60초 이내에 완료되는 단순 요청에 대해 동기 응답을 반환합니다. REST API(동기)를 사용하며, 전체 시나리오 실행은 지원하지 않습니다. 짧은 LLM 단일 호출 등에 적합합니다.

**D-14. SSE 결과 스트리밍**

노드별 진행 상황과 최종 결과를 Server-Sent Events로 스트리밍합니다. 클라이언트에서 SSE 수신·파싱 구현이 필요합니다. B2B 연동 시 최종 결과 추출 규칙을 아래 박스와 같이 합의합니다.

**D-15. 헬스체크**

시스템 및 의존 서비스 상태를 확인합니다. REST API를 통해 모니터링·알림 시스템과 연동할 수 있습니다. 정기 헬스체크로 장애를 조기에 감지합니다.

**D-16. 워커 큐 관리**

LLM, RAG, 웹검색, MCP 등 비동기 작업을 큐에 적재하고 워커가 처리합니다. 내부 처리 방식이며, 워커 수·리소스 스케일링이 처리량에 영향을 줍니다. 피크 부하 시 큐 적체 모니터링이 필요합니다.

**D-17. 실행 로그**

노드별 실행 이력을 데이터베이스에 기록합니다. 감사·디버깅·성능 분석에 활용합니다. 민감 데이터 마스킹 정책을 로그 보관 정책과 함께 정의합니다.

**SSE 최종 결과 규칙**

- 스트림에서 공백 줄 제외 마지막에서 두 번째 줄 = 최종 결과
- 마지막 줄 = 작업 완료 신호

## 6. LLM·RAG·웹검색

LLM·RAG·웹검색 기능은 AI 워크플로의 핵심 지능 계층을 구성합니다. 다중 LLM 프로바이더, 벡터 검색, 외부 검색을 조합하여 질의응답·요약·분석 업무를 수행합니다.

### 주요 기능

**E-01. LLM 모델 등록**

사용할 LLM 프로바이더, 모델명, 파라미터(온도, 최대 토큰 등)를 등록·관리합니다. 관리 UI와 API를 통해 운영 중 모델을 교체하거나 추가할 수 있습니다. 사용 모델·API 키·폐쇄망 정책을 계약·보안 요구에 맞게 설정합니다. 모델별 비용·성능 프로파일을 사전에 비교하는 것이 좋습니다.

**E-02. 다중 프로바이더**

OpenAI, Anthropic, Groq, Upstage, Ollama, Google 등 여러 LLM 프로바이더를 플러그인 아키텍처로 지원합니다. 업무별·노드별로 프로바이더를 선택할 수 있습니다. 프로바이더별 승인·계약·데이터 거주지 정책을 검토해야 합니다. 단일 프로바이더 장애 시 대체 모델 전략도 고려합니다.

**E-03. RAG 노드(실행)**

벡터 데이터베이스에서 유사 문서를 검색하여 LLM 컨텍스트로 제공합니다. REST API를 통해 워크플로 실행 시 호출됩니다. 인덱스·컬렉션 사전 준비와 임베딩 품질 관리가 중요합니다. RAG Builder 모듈과 별도 스택일 수 있으므로 데이터 정합을 협의합니다.

**E-04. 웹검색 노드**

외부 웹 검색 API를 연동하여 최신 정보를 워크플로에 반영합니다. REST API를 통해 검색 결과를 가져옵니다. 외부망 접근과 API 키 발급이 필요합니다. 검색 결과의 활용 범위와 출처 표시 정책을 정의합니다.

**E-05. 임베딩·벡터 저장**

임베딩 결과를 저장할 벡터 데이터 경로를 관리합니다. 파일 시스템 또는 전용 벡터 DB에 저장합니다. 스토리지 용량·백업·복구 정책을 운영 계획에 포함합니다. 대용량 문서 인덱싱 시 디스크 I/O와 검색 지연을 모니터링합니다.

## 7. MCP 연동

MCP(Model Context Protocol) 연동은 AI 에이전트가 데이터베이스·업무 시스템·외부 도구에 직접 접근하게 합니다. 문서 벡터화 없이도 실시간·구조화 데이터 기반 업무 자동화가 가능합니다.

### 주요 기능

**F-01. MCP 서버 등록**

연동할 MCP 서버의 URL과 메타데이터를 등록·관리합니다. 관리 UI와 API를 통해 여러 MCP 서버를 등록할 수 있습니다. 대상 DB·업무 도구용 MCP 서버 구축이 선행 과제입니다. 서버 가용성·인증·네트워크 경로를 사전에 검증합니다.

**F-02. MCP 에이전트 노드**

워크플로 내에서 MCP 에이전트를 실행하여 DB 조회·도구 호출을 수행합니다. 비동기 처리를 지원하는 핵심 연동 방식입니다. DB 직결 검증, 권한 최소화, 쿼리 감사가 도입 시 핵심 검토 항목입니다. RAG와 병행하여 문서+실시간 데이터 통합 시나리오를 구성할 수 있습니다.

**F-03. MCP 노드(동기)**

MCP를 동기 방식으로 직접 호출합니다. 짧고 단순한 도구 호출에 적합합니다. 처리 시간·동시성 제약을 고려하고, 장시간 작업은 에이전트 노드를 사용합니다.

**F-04. MCP 도구 목록 캐시**

등록된 MCP 에이전트의 도구 메타데이터를 캐시하여 에디터·실행 시 빠르게 참조합니다. 내부 캐시를 사용하며, MCP 서버에서 도구가 추가·변경되면 갱신 정책에 따라 반영합니다. 도구 스키마 변경 시 기존 시나리오 호환성을 점검합니다.

**「탈 RAG」 기술적 근거**: MCP를 통해 구조화 데이터·업무 API에 직접 접근하면, 문서 벡터화 없이도 에이전트형 Q&A·업무 처리가 가능합니다. RAG 노드는 보조 수단으로 병행 구성할 수 있습니다.

## 8. RAG Builder (선택 모듈)

RAG Builder는 문서를 수집·변환·분할·벡터화하여 검색 가능한 지식 베이스를 구축하는 선택 모듈입니다. 현재 표준 설치에서는 보류 상태이며, 문서 RAG가 필요한 경우 별도 범위로 협의합니다.

### 주요 기능

**G-01. 문서 변환** *(보류)*

PDF, DOCX, PPT, HWP, TXT 등 다양한 형식의 문서를 Markdown으로 변환합니다. REST API를 통해 변환 파이프라인에 연동합니다. 문서 RAG 도입 시 필수 단계이며, 변환 품질이 검색 정확도에 직접 영향을 줍니다.

**G-02. 문서 분할·벡터화** *(보류)*

변환된 문서를 청크로 분할하고 임베딩을 생성하여 벡터 DB에 저장합니다. REST API를 통해 일괄·증분 처리가 가능합니다. 벡터 DB(Chroma, Milvus, Qdrant 등) 선택과 청크 전략을 사전에 정의합니다.

**G-03. 컬렉션 관리** *(보류)*

벡터 컬렉션을 생성·조회·수정·삭제하고 메타데이터를 관리합니다. REST API를 통해 업무·부서별 컬렉션 분리가 가능합니다.

**G-04. 스케줄 인덱싱** *(보류)*

문서 변경을 감지하여 자동으로 재인덱싱합니다. 스케줄러 기반으로 운영 자동화를 지원합니다. 문서 원본 저장소와 동기화 주기·실패 알림을 정의합니다.

**G-05. RAG Manager UI** *(보류)*

벡터 컬렉션 대시보드, 검색 테스트, 설정을 관리하는 UI를 제공합니다. 별도 설치·포트가 필요할 수 있습니다. 운영 담당자의 인덱스 품질 점검·트러블슈팅에 활용합니다.

**G-06. 워크플로 통합** *(보류)*

RAG Builder에서 생성한 메타데이터를 워크플로 RAG 노드와 동기화합니다. 내부 연동을 통해 검색 대상 컬렉션 정합을 유지합니다. RAG Builder와 워크플로 RAG 노드가 별도 벡터 스택인 경우 데이터 불일치에 주의합니다.

> **주의**: RAG Builder와 워크플로 RAG 노드는 별도 벡터 스택입니다. 문서 RAG 도입 시 범위·데이터 정합을 별도 협의합니다.

## 9. 통합 로그

통합 로그 기능은 플랫폼 전반의 애플리케이션 로그와 감사(audit) 이벤트를 중앙에서 수집·저장·조회합니다. 금융·공공 등 감사 요구가 높은 환경에 적합합니다.

### 주요 기능

**H-01. 로그 배치 수집**

여러 애플리케이션에서 발생한 로그를 배치로 수집합니다. REST API(비동기)를 통해 대량 이벤트를 효율적으로 전달합니다. 전 앱 SDK 연동과 수집 실패 시 재시도·버퍼 정책을 정의합니다.

**H-02. 로그 단건 수집**

단일 로그 이벤트를 즉시 수집합니다. REST API를 통해 실시간·저빈도 이벤트에 사용합니다. 배치 수집과 용도에 따라 선택합니다.

**H-03. 로그 ID 생성**

서버에서 유니크한 로그 ID를 부여합니다. 분산 환경에서도 추적·상관관계 분석이 가능합니다. 여러 시스템 로그를 하나의 업무 트랜잭션으로 묶을 때 활용합니다.

**H-04. 감사 로그 라우팅**

로그 타입이 audit인 이벤트를 별도 저장소에 라우팅합니다. 내부 라우팅 정책으로 금융권 등 감사 요구를 충족합니다. 감사 로그의 보관 기간·접근 권한·변조 방지를 정책으로 정의합니다.

**H-05. 일반 로그 저장**

audit이 아닌 일반 로그를 파일 기반으로 저장·순환·백업합니다. 보관 기간·디스크 용량·압축 정책을 운영에 맞게 설정합니다. 대용량 로그 환경에서는 보관 티어(핫·콜드)를 검토합니다.

**H-06. 앱별 로그 레벨 제어**

애플리케이션별로 런타임 로그 레벨(debug, info, warn, error)을 조정합니다. REST API를 통해 재배포 없이 운영 중 디버깅이 가능합니다. 프로덕션에서 debug 레벨 활성화 시 민감 정보 노출에 주의합니다.

**H-07. 로그 관리 UI**

수집된 로그를 조회·검색·관리하는 대시보드를 제공합니다. 관리 UI를 통해 감사 담당자·운영자가 접근합니다. 역할별 조회 권한과 export 정책을 정의합니다.

**H-08. 로그 접근 제어**

애플리케이션이 감사 로그 저장소에 직접 접근하지 못하도록 제한합니다. 수집 API(ingest)만을 통한 쓰기를 정책으로 강제합니다. 보안·분리 원칙에 따라 tamper-evident 보관을 권장합니다.

**로그 envelope 필수 필드**: 로그 ID, 타임스탬프, 앱 식별자, 로그 타입, 레벨, 메시지

## 10. 외부 시스템 연동

외부 시스템 연동 기능은 기업 내 업무 애플리케이션·포털·챗봇 등에서 DeepCube.AI의 AI 시나리오를 REST API로 호출할 수 있게 합니다.

### 10.1 연동 방식 개요

DeepCube.AI는 표준 REST API 기반의 4단계 연동 패턴을 제공합니다:

1. **인증 토큰 발급** (API Key 기반)
2. **사용 가능한 AI 시나리오 목록 조회**
3. **시나리오 실행 요청**
4. **실시간 결과 수신** (Server-Sent Events)

### 10.2 연동 기능

**API Key 관리**

외부 시스템별로 연동용 API Key를 발급·관리합니다. 키 보관·회전·폐기 정책을 보안 절차에 포함해야 합니다. 시스템별 키 분리로 유출 시 영향 범위를 제한할 수 있습니다. 키는 환경 변수·시크릿 매니저에 저장하고 소스 코드에 포함하지 않습니다.

**토큰 기반 인증**

발급된 API Key를 JWT 토큰으로 교환하여 이후 API 호출에 사용합니다. 토큰 만료·재발급 주기를 연동 클라이언트에 맞게 구현합니다. 만료 전 자동 갱신 로직을 두면 장시간 세션 연동이 안정적입니다.

**시나리오 원격 실행**

REST API를 통해 외부 앱에서 AI 워크플로를 비동기 실행합니다. 응답 시간·동시 호출 수·타임아웃을 SLA에 맞게 설계합니다. 피크 시간대 부하 분산·재시도 정책도 함께 정의합니다.

**실시간 결과 스트리밍**

SSE(Server-Sent Events)로 시나리오 실행의 단계별 진행과 최종 결과를 전송합니다. 클라이언트에서 SSE 수신·파싱·최종 결과 추출을 구현해야 합니다. 언어·프레임워크별 SSE 라이브러리 호환성을 사전에 검증합니다.

**파일 연동**

입력 파일 업로드와 결과 파일 다운로드를 지원합니다. 로컬·SFTP·객체 스토리지 등 스토리지 옵션은 도입 시 협의합니다. 대용량 파일·바이러스 스캔·보관 기간 정책을 함께 정의합니다.
